# Risk Audit Table — Policy Copilot

This table documents the failure modes, consequences, detection mechanisms, mitigations, and residual risks for an audit-ready policy compliance assistant. It complements the project risk register (Table 2.2 in the dissertation report) with a system-level threat model focused on the risks that matter most in a compliance/policy context.

| # | Failure Mode | Consequence | Detection / Logging | Mitigation | Residual Risk |
|---|-------------|-------------|-------------------|------------|---------------|
| R1 | **Hallucination** — LLM generates claims not present in retrieved evidence | User acts on fabricated policy guidance; legal/compliance exposure | Claim verification pipeline flags ungrounded claims; `ungrounded_rate` metric in evaluation; audit export records per-claim support status | Post-generation claim splitting + NLI-based verification (B3 pipeline); unsupported claims surfaced in Audit Trace; abstention when support rate falls below threshold | Verification relies on NLI model accuracy; adversarial or subtle hallucinations may evade detection |
| R2 | **Citation fabrication** — system cites a paragraph that does not support the claim | False sense of evidential grounding; user trusts answer based on incorrect citation | Citation precision/recall metrics against golden-set labels; Audit Trace displays cited paragraph text for manual inspection | Citation enforcement requires each claim to map to a retrieved passage; citation metrics evaluated per-baseline; evidence rail shows full quoted text | Citation correctness depends on claim-splitting granularity and NLI threshold calibration |
| R3 | **Contradiction suppression** — conflicting evidence exists but system does not surface it | User receives one-sided answer on a genuinely contested policy question | Contradiction detection module using NLI entailment/contradiction labels; `contradiction_recall` and `contradiction_precision` metrics | Dedicated contradiction detection stage in B3 pipeline; side-by-side conflict display in Audit Trace and HTML export; abstention option when contradictions are detected | NLI models may miss subtle or domain-specific contradictions; threshold choice trades recall against false positives |
| R4 | **Abstention failure** — system answers when it should abstain (or abstains when it should answer) | Over-answering: user gets unreliable guidance on out-of-scope queries. Under-answering: system is unhelpful on valid queries | `abstention_accuracy` metric on unanswerable golden-set slice; confidence threshold logged per query; abstention reason recorded in QueryResult | Two-stage abstention: pre-generation confidence check on reranker scores, post-generation support-rate check; explicit "insufficient evidence" UI banner | Calibration of confidence threshold is corpus-dependent; no guarantee of optimal abstention boundary |
| R5 | **Stale policy corpus** — indexed documents are outdated relative to current regulations | User receives guidance based on superseded policy; compliance risk | Corpus metadata includes document dates; `verify_artifacts.py` logs corpus provenance; run manifests record corpus version | Corpus is versioned and frozen for evaluation reproducibility; system does not claim real-time policy updates; limitations section explicitly states corpus staleness risk | No automated staleness detection; user must verify currency of policy documents independently |
| R6 | **Adversarial prompt injection** — malicious input manipulates system behaviour | System produces harmful, misleading, or policy-violating output; jailbreak of safety constraints | Query logged in audit trail; unusual output patterns detectable in review workflow | Input is passed through retrieval pipeline (not direct LLM prompting in B2/B3); system prompt includes instruction-following constraints; no user-uploaded documents in current scope | No dedicated adversarial input filter; LLM-level prompt injection defences depend on provider guardrails |
| R7 | **Backend fallback without disclosure** — system silently uses BM25 when dense retrieval was requested | Provenance claim is misleading; evaluation results may not reflect stated configuration | `backend_requested` vs `backend_used` fields logged in every run output; `verify_artifacts.py` warns on mismatch; strict mode fails on mismatch | Explicit fallback logging in retrieval layer; UI Audit Trace shows both requested and used backend; provenance warnings in verification script | Fallback is by design for environments without ML dependencies; the risk is mislabelling, not the fallback itself |
| R8 | **Automation bias / overreliance** — user trusts system output without critical review | Incorrect policy decisions made without human judgement; accountability gap | Reviewer Mode enables structured human scoring; abstention and contradiction banners signal uncertainty | Explicit uncertainty communication: abstention banners, contradiction warnings, confidence indicators, evidence rail with support status; system never claims to replace legal/policy expertise | No user study conducted to measure actual overreliance; mitigation is design-level, not empirically validated |
| R9 | **Privacy / data leakage** — sensitive content exposed through system outputs or logs | Breach of data protection obligations; reputational and legal consequences | No personal data in the evaluation corpus (public university handbook); audit logs stored locally | Evaluation uses only publicly available policy documents; no PII in golden set or test queries; local-first architecture (no cloud telemetry); API keys managed via environment variables | If deployed on sensitive corpora, additional access controls and data handling procedures would be required |
| R10 | **Environmental / cost impact** — LLM API calls consume energy and incur financial cost | Carbon footprint of inference; unexpected API costs during evaluation | Token usage and latency tracked per query in run artifacts; cost estimation available in evaluation summary | Extractive fallback mode (B2) operates without LLM calls; offline reproduction uses cached results; evaluation harness supports cost-aware configuration selection | No carbon offsetting; cost depends on provider pricing; large-scale deployment would require cost monitoring |
